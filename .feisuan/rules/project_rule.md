## 三、开发规范

### 1. 命名规范

| 类型       | 命名方式             | 示例                  |
|------------|----------------------|-----------------------|
| 类名       | UpperCamelCase       | `PetManager`          |
| 方法/变量  | lowerCamelCase       | `feedPet()`           |
| 常量       | UPPER_SNAKE_CASE     | `MAX_HUNGER`          |
| 文件名     | kebab-case           | `pet-data.json`       |

### 2. 代码组织规范

- **包结构**：
  - `app`：应用入口和主程序
  - `core`：核心业务逻辑
  - `data`：数据处理相关功能
  - `renderer`：UI渲染相关
  - `types`：类型定义和接口
  - `locales`：多语言支持

- **模块化原则**：
  - 每个功能模块应独立，低耦合
  - 数据处理与业务逻辑分离
  - 渲染逻辑与业务逻辑分离

### 3. 数据处理规范

- **JSON数据处理**：
  - 所有JSON文件应放在 `src/data/json/` 目录下
  - 使用统一的JSON格式规范
  - 数据修改后应有版本控制机制

- **数据文件组织**：
  - 游戏数据分类存放（成就、冒险、物品、任务、学习）
  - 每个数据类别有独立的子目录

### 4. 资源管理规范

- **资源文件**：
  - 所有资源文件（图片、音频等）放在 `assets/` 目录
  - 资源文件按功能分类子目录
  - 资源文件命名应具有描述性

### 5. 国际化支持

- **多语言资源**：
  - 所有语言文件放在 `src/locales/` 目录
  - 使用标准化的语言代码（如 `en-US`, `zh-CN`）
  - 语言文件采用JSON或属性文件格式

## 四、编码原则

| 原则       | 说明                                       |
|------------|--------------------------------------------|
| **SOLID**  | 高内聚、低耦合，增强可维护性与可扩展性     |
| **DRY**    | 避免重复代码，提高复用性                   |
| **KISS**   | 保持代码简洁易懂                           |
| **YAGNI**  | 不实现当前不需要的功能                     |
| **OWASP**  | 防范常见安全漏洞，如输入验证、XSS等        |

## 五、代码风格规范

### 1. 注释规范

- 所有公共类、方法必须有Javadoc注释
- 注释应使用中文编写，便于团队理解
- 复杂逻辑应有行内注释说明

### 2. 日志规范

- 使用结构化日志记录
- 重要操作和错误必须记录日志
- 日志应包含足够的上下文信息

### 3. 错误处理规范

- 所有异常应有适当的处理
- 提供用户友好的错误信息
- 记录详细的错误日志用于调试

## 六、版本控制规范

- 使用语义化版本号（MAJOR.MINOR.PATCH）
- 提交信息应清晰描述变更内容
- 重要功能应创建独立的分支开发
